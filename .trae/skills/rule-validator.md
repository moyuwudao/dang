# Rule Validator - 规则验证器

## 核心理念

**规则是约束，也是保护。**
Rule Validator 确保每个规则文件都符合规范，避免：
- 缺少文件头导致规则无法识别
- 未注册 INDEX.md 导致规则无法发现
- 格式不规范导致规则无效

---

## 触发条件

### 自动触发
- 在 `.trae/rules/` 目录下创建新 `.md` 文件
- 修改现有规则文件

### 手动触发
```
/validate-rule <文件名>
```

---

## 验证流程

### 步骤 1：文件头验证

**检查内容**：
```
□ 文件以 --- 开头
□ 包含 alwaysApply: true | false
□ description 不为空
□ description 不超过 50 字
```

**验证结果**：
- ✅ 文件头正确
- ❌ 缺少文件头 → 生成标准文件头建议
- ❌ alwaysApply 未设置 → 提示设置 true/false
- ❌ description 为空 → 提示填写描述
- ❌ description 超过 50 字 → 提示精简

### 步骤 2：命名规范验证

**检查内容**：
```
□ 核心规则：大写驼峰（SOUL.md）
□ 功能规则：大写下划线（API_DESIGN.md）
□ 专项规则：项目名-功能（dang-构建apk规则.md）
□ API规范：API名_功能_API（QWEN_ASR_API.md）
```

**验证结果**：
- ✅ 命名规范
- ❌ 命名不规范 → 提示正确命名格式

### 步骤 3：内容结构验证

**检查内容**：
```
□ 包含 # 标题（与文件名对应）
□ 包含 ## 概述
□ 包含 ## 核心内容
□ 包含 ## 更新记录
```

**验证结果**：
- ✅ 内容结构完整
- ❌ 缺少标题 → 提示添加
- ❌ 缺少概述 → 提示添加
- ❌ 缺少更新记录 → 提示添加

### 步骤 4：INDEX.md 注册验证（最关键）

**检查内容**：
```
□ 在 INDEX.md 中存在该规则
□ 标注了 alwaysApply 状态
□ 有 ✅ 标记
□ 规则总数正确
```

**验证结果**：
- ✅ 已注册
- ❌ 未注册 → **自动修复**：添加到对应分类
- ❌ 总数不匹配 → **自动修复**：更新计数

### 步骤 5：更新日志验证

**检查内容**：
```
□ INDEX.md 中有该规则的更新记录
□ 包含日期、更新内容、影响
```

**验证结果**：
- ✅ 更新日志完整
- ❌ 缺少更新日志 → **自动修复**：添加更新记录

## 输出格式

### 验证通过
```
✅ 规则验证通过

验证项目：
✅ 文件头正确
✅ 命名规范
✅ 内容结构完整
✅ INDEX.md 已注册
✅ 更新日志完整

该规则符合规范，可以正常使用。
```

### 验证失败
```
❌ 规则验证失败

验证项目：
✅ 文件头正确
✅ 命名规范
❌ 内容结构不完整
   - 缺少 ## 更新记录
❌ INDEX.md 未注册
   - 已自动修复：添加到 dang 项目特定规则
✅ 更新日志完整

请根据提示修复问题，然后重新验证。
```

## 自动修复功能

### 可自动修复的问题

| 问题 | 修复方式 |
|-----|---------|
| 未在 INDEX.md 注册 | 自动添加到对应分类 |
| 规则总数不匹配 | 自动更新计数 |
| 缺少更新日志 | 自动添加更新记录 |

### 需要手动修复的问题

| 问题 | 修复方式 |
|-----|---------|
| 缺少文件头 | 手动添加 YAML Front Matter |
| alwaysApply 未设置 | 手动设置 true/false |
| description 为空 | 手动填写描述 |
| 命名不规范 | 手动重命名文件 |
| 内容结构不完整 | 手动补充缺失部分 |

### 步骤 6：交叉引用验证（新增）

**检查内容**：
```
□ 规则中引用的 [OTHER_RULE.md] 目标文件存在
□ 引用的案例编号（CASE-XXX）格式正确
□ 引用的文件路径格式正确
```

**验证结果**：
- ✅ 交叉引用正确
- ❌ 引用目标不存在 → 列出不存在的引用
- ❌ 案例编号格式错误 → 提示正确格式

### 步骤 7：案例连续性检查（Troubleshooting 文件专用）

**检查内容**：
```
□ CASE 编号连续（无跳号）
□ 索引表与实际案例内容一致
□ 每个案例包含完整的四个部分
```

**验证结果**：
- ✅ 案例编号连续
- ❌ 跳号（如 CASE-001, CASE-003）→ 提示跳号位置
- ❌ 索引表缺少案例 → 自动修复：同步索引表

### 步骤 8：alwaysApply 一致性检查（新增）

**检查内容**：
```
□ alwaysApply: true 的规则是否为核心规则
□ alwaysApply: false 的规则是否适合按需加载
□ alwaysApply: true 的规则数量是否合理（建议 ≤ 7 个）
```

**验证结果**：
- ✅ alwaysApply 设置合理
- ⚠️ 建议评估 → 列出可能应调整的规则
- ❌ alwaysApply: true 过多 → 建议精简

## 批量验证

### 验证所有规则
```
/validate-all-rules
```

```

✅ 通过 X 个
❌ 失败 Y 个
⚠️ 警告 Z 个

失败文件：
- file1.md: INDEX.md 未注册

```

## 集成到工作流

### 创建规则时的标准流程

```
1. 创建规则文件
   ↓
2. 编写规则内容
   ↓
3. 运行 /validate-rule <文件名>
   ↓
4. 根据提示修复问题
   ↓
5. 重新验证直到通过
   ↓
6. 完成！
```

### 与 RULES_GUIDE.md 的关系

RULES_GUIDE.md 是**规范文档**，Rule Validator 是**验证工具**。
- RULES_GUIDE.md 告诉你"应该怎么做"
- Rule Validator 检查"你是否做到了"

### 方案C 追加能力

方案C重构后新增的验证能力：
- 🔍 交叉引用验证：确保规则间引用关系正确
- 📊 案例连续性检查：Troubleshooting 文档的 CASE 编号管理
- ⚖️ alwaysApply 一致性：确保核心规则数量合理
- 📦 批量验证：一次性验证所有规则
