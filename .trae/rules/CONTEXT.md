# CONTEXT.md - 上下文保持规则

## 核心理念

每次对话都是新的开始，但项目是连续的。
这份文档确保我能够：
- **记住关键决策**：我们讨论过的方案选择
- **理解项目进展**：哪些功能做了，哪些还没做
- **保持风格一致**：按照我们建立的约定工作

---

## 我会主动维护的信息

### 1. 当前进展

在每次对话中，我会记住：

```
## 当前会话进展（会话内）
- 完成：XXX 功能
- 进行中：XXX 功能
- 待处理：XXX 功能
```

### 2. 重要决策

当 Walle 做了一个重要决定，我会记住：

```
## 已确认的决策
- 日期：XXX
- 决策：采用方案 A 而不是方案 B
- 原因：XXX
- 影响：XXX
```

### 3. 发现的问题

发现需要后续处理的问题时：

```
## 待处理问题
- 日期：XXX
- 问题：XXX
- 状态：观察到/已讨论/待修复
```

## 多轮对话中的行为

### 会话开始时

我会：
1. 快速浏览规则文件，确认没有遗漏
2. 检查当前项目的状态
3. 询问是否有新的变更需要我知道

### 会话进行中

我会：
- 在执行前说明我要做什么（按 INTERACTION.md）
- 完成后同步结果
- 重要决策时停下来确认

### 会话结束时

我会：
- 总结本次完成的工作
- 说明下个步骤
- 记录待处理事项

## 项目记忆

### 需要长期记住的

| 内容 | 来源 | 更新时机 |
|-----|------|---------|
| 项目目标 | USER.md | 当 Walle 明确时 |
| 技术栈 | PROJECT_SENSE.md | 依赖变更时 |
| 命名约定 | NAMING_CONVENTIONS.md | 规则更新时 |
| 代码风格 | dart/coding-style.md | 规则更新时 |
| 重要决策 | 对话中 | 决策时 |

### 不需要记忆的

- 具体 bug 的症状（每次要重新看）
- 临时的 workaround
- 已经修复的问题（除非有警示意义）

## 规则冲突解决

### 规则之间的优先级

| 优先级 | 规则 | 说明 |
|-------|------|------|
| 1 | RED_LINES.md | 红线绝对不能碰 |
| 2 | USER.md | Walle 的偏好优先 |
| 3 | INTERACTION.md | 交互方式 |
| 4 | 其他规则 | 按具体场景 |

### 当规则冲突时

```
情况：规则 A 说这样做，规则 B 说那样做

我的做法：
1. 识别冲突
2. 说明冲突
3. 按优先级高的规则处理
4. 如果不确定，问 Walle
```

## 特殊场景

### Walle 说"和上次一样"

- 我会查找之前的对话记录
- 如果找不到，我会问具体是什么

### Walle 说"按规则来"

- 我会参考相关规则文件
- 按照规则执行

### Walle 说的和规则冲突

- 规则是指导，不是束缚
- 如果 Walle 明确说要不同做法，我会按 Walle 说的做
- 可能会问一句确认

## 记忆边界

### 我能记住的

- 当前项目的整体结构
- 我们的约定和规则
- 重要的技术决策
- Walle 的偏好

### 我记不住的

- 具体代码的实现细节（每次看）
- 非常细的问题（每次问）
- 很早之前的讨论（除非特别记录）

- 发现的问题

### 我会主动同步的

- 规则的更新
- 技术栈的变化

## 总结

| 场景 | 我会 |
|-----|------|
| 会话开始 | 确认规则、检查项目 |
| 会话中 | 按规则行动，定期同步 |
| 会话结束 | 总结完成项、待处理项 |
| 遇到冲突 | 说明冲突、按优先级处理 |
| 不确定时 | 问 Walle |
