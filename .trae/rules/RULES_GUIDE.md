# RULES_GUIDE.md - 规则编写指导

## 核心理念

规则是给 AI 用的，不是给人看的。
- **精准 > 完整** — 模糊的规则不生效
- **可验证 > 可描述** — 能检查的才是好规则
- **引用 > 复制** — 避免内容重复

---

## 规则文件标准格式

```markdown
---
alwaysApply: true | false
description: 简短描述，不超过50字
---

# 标题

## 概述
## 核心内容
## 更新记录
```

### alwaysApply 设置

| 设置 | 适用 | 示例 |
|-----|------|------|
| true | 每次对话都需要 | SOUL, USER, INTERACTION, RED_LINES |
| false | 特定场景才需要 | BUILD, API_*, SERVER_* |

### 命名规范

| 类型 | 格式 | 示例 |
|-----|------|------|
| 核心规则 | 大写驼峰 | SOUL.md, RED_LINES.md |
| 功能规则 | 大写下划线 | API_DESIGN.md, BUILD.md |
| 专项规则 | 项目名-功能 | dang-构建apk规则.md |
| API 规范 | API名_功能_API | QWEN_ASR_API.md |

## 规则编写 checklist

### 创建新规则
- [ ] YAML Front Matter（alwaysApply + description）
- [ ] 命名符合规范
- [ ] 内容结构完整（标题、概述、内容、更新记录）
- [ ] 在 INDEX.md 注册
- [ ] 检查是否与现有规则冲突
- [ ] **防重复检查**：新规则是否引入了"多做一次"的额外操作？是否与现有规则有重叠内容？

### 更新现有规则
- [ ] 更新内容
- [ ] 更新更新记录
- [ ] 检查影响范围

### 防重复编写原则

| 原则 | 说明 |
|------|------|
| **一个真相来源** | 每个知识点只在一处定义，其他地方只引用 |
| **不要双重肯定** | 不要说"A 很重要，请务必做"又说"B 中也要做 A" |
| **上限明确** | 任何"反复/多次/持续"类的描述必须带上限数字 |
| **避免触发链** | 规则 A 触发规则 B，规则 B 又触发规则 C → 禁止 |
| **失败闭环** | 任何操作如果可能失败，规则中必须写清楚"失败 N 次后怎么做" |

## 规则优先级

项目特定规则 > 语言特定规则 > 通用规则

---

## 规则引用规范

引用其他规则，而不是复制内容：
```markdown
详见 [BUILD.md](BUILD.md)
参考 CASE-001: [APK 复制后不是最新](BUILD_TROUBLESHOOTING.md#case-001)
```

---

## 规则维护流程

### 新增
1. 确定类型和命名
2. 编写内容
3. INDEX.md 注册
4. 检查冲突

### 修改
1. 修改内容
2. 更新记录
3. 检查影响

### 删除
1. 确认无引用
2. 从 INDEX.md 移除
3. 删除文件
